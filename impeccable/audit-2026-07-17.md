## Detailed Findings by Severity

---

### [P1] Clickable divs lock out keyboard users
- **Location:** `index.html:717` (`.hrun` history cards), `index.html:193–200` (`.ocollapse-head`)
- **Category:** Accessibility
- **Impact:** Keyboard-only users and screen reader users cannot expand history entries or scraped content sections. These are `div` elements with click handlers but no `role`, no `tabindex`, and no `keydown` handler. The entire History tab's interactive behavior is inaccessible.
- **WCAG:** 2.1.1 Keyboard (Level A), 4.1.2 Name, Role, Value (Level A)
- **Recommendation:** Convert `.ocollapse-head` to a `<button>` or add `role="button" tabindex="0"` + `keydown` handler for Enter/Space. For `.hrun` cards: either make the whole card a `<button>` (with the delete button nested inside, separated by pointer-events), or add `tabindex="0" role="button"` + keyboard handler.
- **Suggested command:** `/impeccable harden`

---

### [P1] Tab widget missing ARIA roles
- **Location:** `index.html:244–248` (`.tabs`), `index.html:251–355` (panels)
- **Category:** Accessibility
- **Impact:** Screen readers treat the tab row as a list of ordinary buttons with no announced relationship to the panels they control. The active state is visually communicated only through CSS, not semantics.
- **WCAG:** 4.1.2 Name, Role, Value (Level A)
- **Recommendation:**
  - Add `role="tablist"` to `.tabs`
  - Add `role="tab"` and `aria-selected="true/false"` to each `.tab` button; update `aria-selected` on click
  - Add `role="tabpanel"` and `aria-labelledby="<tab-id>"` to each `.panel` section
  - Implement arrow-key navigation between tabs (standard tab widget keyboard contract)
- **Suggested command:** `/impeccable harden`

---

### [P1] Ink Dim (`#52655c`) fails WCAG AA contrast
- **Location:** `index.html:83` (`::placeholder`), `index.html:120` (`.step .st`), `index.html:168` (`.ohead .osize`)
- **Category:** Accessibility
- **Impact:** Placeholder text, step timestamps, and output size labels all render at approximately 2.7:1 contrast on their respective dark backgrounds (Forest Void and Forest Panel). WCAG AA requires 4.5:1 for normal text.
- **WCAG:** 1.4.3 Contrast Minimum (Level AA)
- **Recommendation:** Raise `#52655c` to a value that reaches ≥4.5:1 on `#0e1512` and `#141d19`. A value around `#748b82` or lighter would clear the threshold. Define it as `--ink-dim` in the `:root` token block so all instances update together.
- **Suggested command:** `/impeccable harden`

---

### [P1] Status regions have no `aria-live`
- **Location:** `index.html:289` (`#searchStatus`), `index.html:340` (`#fetchStatus`), `index.html:353` (`#historyStatus`), `index.html:241` (`#health`)
- **Category:** Accessibility
- **Impact:** Search progress steps, fetch progress, history loading states, and gateway health status all update dynamically via JS but have no `aria-live` or `role="status"` attribute. Screen reader users receive no announcement of any operation's progress or completion.
- **WCAG:** 4.1.3 Status Messages (Level AA)
- **Recommendation:** Add `aria-live="polite"` to `.status` elements and `role="status"` to `#health`. For the step log, `aria-live="polite"` on the `.steps` container is sufficient — each appended step will be announced.
- **Suggested command:** `/impeccable harden`

---

### [P2] External CDN dependencies in a self-hosted tool
- **Location:** `index.html:7–10`
- **Category:** Performance
- **Impact:** Google Fonts and `unpkg.com/lucide@latest` are loaded from public CDNs. This is ironic for a self-hosted privacy-first tool: every session leaks a connection to Google and Cloudflare/unpkg. It also creates a reliability dependency — the app breaks visually if these CDNs are unreachable. The `@latest` Lucide URL is not versioned, making it non-cacheable long-term and vulnerable to upstream breaking changes.
- **Recommendation:** Self-host both assets. For Lucide: `npm install lucide` (or grab the UMD build) and serve from `/static/`. For Google Fonts: download the woff2 files and add `@font-face` declarations inline or in a local static file.
- **Suggested command:** `/impeccable optimize`

---

### [P2] Touch targets below 44×44px
- **Location:** `index.html:91–97` (`.pill`), `index.html:137` (`.hdel`), `index.html:170–175` (`.obtn`)
- **Category:** Responsive Design
- **Impact:** Pills compute to approximately 26px height (`5+5px padding + 16px line-height`). The delete button (`.hdel`) is approximately 24px. The output panel copy/download buttons (`.obtn`) are approximately 26px. On touch devices, these targets require precise tapping and are prone to misactivation.
- **WCAG:** 2.5.8 Target Size Minimum (Level AA, WCAG 2.2) requires 24×24px; 2.5.5 (Level AAA) recommends 44×44px.
- **Recommendation:** Add a `@media (max-width: 600px)` block increasing pill padding to `8px 14px` (height ≈ 34px) and `.hdel` padding to `6px 10px; min-height: 36px`.
- **Suggested command:** `/impeccable adapt`

### [P2] Missing token definitions for Ink Dim and Forest Deep
- **Location:** `index.html:11–27` (`:root`), used at lines 83, 120, 136, 168, 183, 209, 214–215
- **Category:** Theming
- **Impact:** The DESIGN.md documents `Ink Dim (#52655c)` and `Forest Deep (#0c110f)` as named palette roles, but neither is defined as a CSS custom property. The values appear 4–6 times each as hardcoded hex strings. Changing either value (e.g., to fix the contrast issue above) requires find-and-replace rather than a single token edit.
- **Recommendation:** Add to `:root`:
  ```css
  --deep: #0c110f;
  --ink-dim: #52655c;   /* will need to be raised — see P1 contrast finding */
  ```
  Then replace all inline `#0c110f` and `#52655c` references. Also note `#b9c7bf` (snippet text) and `#cdd9d2` (prose/raw text) appear 2–3 times each with no token — consider `--ink-sub` for these.
- **Suggested command:** `/impeccable harden`

---

### [P2] Missing focus styles on pills and tabs
- **Location:** `index.html:61–97` (`.tab`, `.pill`)
- **Category:** Accessibility
- **Impact:** The DESIGN.md specifies `0 0 0 3px rgba(245,184,67,.12)` as the universal focus treatment. Inputs implement this correctly. Pills and tabs do not — they rely on browser-default outlines, which are visually jarring against the dark surface and inconsistent with the design system.
- **Recommendation:** Add to `.tab` and `.pill`:
  ```css
  .tab:focus-visible, .pill:focus-visible {
    outline: none;
    box-shadow: 0 0 0 3px rgba(245,184,67,.12);
  }
  ```
- **Suggested command:** `/impeccable harden`

---

### [P2] No mobile layout breakpoints
- **Location:** `index.html:46–228` (layout CSS)
- **Category:** Responsive Design
- **Impact:** The layout relies entirely on `flex-wrap` behavior. On small screens (< 400px): the search `.row` (query input + Search button) will shrink the input without stacking; options rows wrap unpredictably; the tab border-radius `0 12px 12px 12px` assumes the first tab is always top-left, which breaks if tabs wrap.
- **Recommendation:** Add a `@media (max-width: 600px)` block: stack `.row` to `flex-direction: column`, make `.go` button full-width, increase pill padding for touch.
- **Suggested command:** `/impeccable adapt`

---

### [P3] Delete button accessible name unreliable
- **Location:** `index.html:723` (`.hdel` button)
- **Category:** Accessibility
- **Impact:** `title="delete run"` provides an accessible name in most screen readers but is not universally reliable. `aria-label` is the explicit, reliable alternative.
- **Recommendation:** Add `aria-label="delete run"` alongside `title="delete run"`.
- **Suggested command:** `/impeccable harden`

---

### [P3] Inline styles scattered throughout HTML
- **Location:** `index.html:254, 258, 263, 280, 290, 315–337` and ~15 more instances
- **Category:** Theming
- **Impact:** `style="flex:1"`, `style="align-self:end"`, `style="display:none"`, `style="text-transform:none;letter-spacing:0"` etc. appear on ~20 elements. These override the CSS cascade, resist refactoring, and make the design system harder to reason about.
- **Recommendation:** Extract repeated patterns into utility classes. The one-off responsive widths can stay inline; repeated layout patterns should become classes.
- **Suggested command:** `/impeccable polish`

---

### [P3] History detail uses `<details>`/`<summary>` with inline styles
- **Location:** `index.html:751` (inside `loadHistory`)
- **Category:** Accessibility / Theming
- **Impact:** `<summary style="cursor:pointer;color:var(--amber-dim);font-size:12px">` inherits no focus treatment from the design system. The `<details>` open/close animation is browser-native and inconsistent with the `rise` / transition patterns elsewhere.
- **Recommendation:** Replace the `<details>` block with the `.ocollapse` pattern used for scraped output, once that pattern is fixed with keyboard support (see P1 finding). This unifies the pattern and applies the design system's focus treatment.
- **Suggested command:** `/impeccable harden`

## Patterns & Systemic Issues

1. **ARIA is treated as optional.** Every dynamic interaction (tab switching, collapses, status updates, history expansion) updates the DOM correctly but omits the ARIA layer entirely. A single `/impeccable harden` pass targeting ARIA + keyboard would resolve all four P1 accessibility findings at once.

2. **Token system covers 70% of the palette.** The primary surface, accent, and semantic colors are tokenized well. The secondary ink values and Forest Deep are not. Every hardcoded hex in the lower ink range will need manual tracking if the palette shifts. The fix is four new lines in `:root`.

3. **Touch targets are consistently underbuilt.** Every small interactive element (pills, delete, copy, download, view-toggle buttons) lands in the 24–28px height range. A single breakpoint rule could lift all of these at once for mobile.

---

## Positive Findings

- **Color semantics are airtight.** Harvest appears exactly where DESIGN.md says it should. Phosphor is only used for live/success states. Fault red is only used for errors. This discipline is rare and makes the UI readable at a glance.
- **Animation is purposeful and efficient.** The `rise` entrance uses `opacity + transform` only (compositable, GPU-friendly). The `pulse` dot and `spin` icon are CSS animations. No layout properties animated anywhere.
- **The font domain rule is enforced perfectly.** Fraunces appears only on result titles, the fetched document title, and the h1 wordmark. Every UI label, button, and meta line is Space Mono. No crossover.
- **XSS prevention is solid.** The `esc()` function is applied consistently to all user-sourced and API-sourced content before innerHTML insertion. The markdown renderer escapes first, then replaces — correct order.
- **Token usage for primary values is consistent.** `--amber`, `--green`, `--red`, `--panel`, `--panel-2`, `--bg`, `--muted`, `--line` are used everywhere they should be, with no visible deviations.
- **The scanline texture stays on the body.** No nested panels or cards accidentally receive the atmospheric treatment — the `body::before` with `position: fixed` is correctly scoped.
